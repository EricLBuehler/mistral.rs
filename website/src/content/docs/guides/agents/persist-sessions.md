Agent conversations accumulate state. The message history, the tool calls that have already run, the Python variables defined in a code-execution session: all of those are meaningful context that the model will want to reuse on the next turn. mistralrs holds this state in memory on the server, keyed by a session id, and makes it available to any request that provides that id.

This guide covers how session matching works, how to export and import sessions (for persistence across server restarts or migrations between servers), and what the session lifetime looks like.

## How sessions are identified

Every agentic request gets a session id. Two cases:

**You did not pass one.** The server generates a fresh UUID, runs the request against an empty session state, and returns the new id in the response (as `session_id` in the JSON body, and as part of the final streaming chunk if you are streaming).

**You passed an explicit id.** The server looks up that session. If it exists, the request picks up where the previous one left off. If it does not exist, a new session is created under that id.

Passing a session id looks like this:

```bash
curl http://localhost:1234/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "default",
    "messages": [{"role": "user", "content": "What is 2 to the 10th?"}],
    "session_id": "user-42-chat-abc"
  }'
```

## Implicit matching for clients that do not pass ids

Some clients (older OpenAI SDKs, anything that does not know about `session_id`) cannot pass the field. For those, mistralrs falls back to matching sessions by message prefix. If the messages you send match the start of an existing session's user-visible history, that session is picked up automatically.

This works transparently in most cases. The main failure mode is when two users happen to send the same opening messages: the engine might match them to the same session, which is usually not what you want. When you care about correctness here, pass the session id explicitly.

## Exporting a session

The GET endpoint serializes a session to JSON, including message history, tool calls, and any images or videos that have been passed through:

```bash
curl http://localhost:1234/v1/sessions/user-42-chat-abc
```

The response is a `SerializedSession` object with the full state. Save this to disk or send it over the wire; everything you need to reconstruct the session is in there.

## Importing a session

The PUT endpoint accepts a serialized session and installs it under the given id:

```bash
curl -X PUT http://localhost:1234/v1/sessions/user-42-chat-abc \
  -H "Content-Type: application/json" \
  -d @saved-session.json
```

Subsequent requests with that session id pick up exactly where the exported state left off. This is the primitive for moving sessions between servers, restoring state after a restart, or handing a conversation to a different user.

## Deleting sessions

```bash
curl -X DELETE http://localhost:1234/v1/sessions/user-42-chat-abc
```

Always returns 200, whether or not the session existed. Useful for ending conversations cleanly from a client.

## Lifetime

Sessions live in memory. Three things end a session:

- **Idle expiry**: sessions untouched for 30 minutes are evicted automatically.
- **Capacity**: the server keeps at most 128 sessions; if you create a 129th, the oldest one gets kicked out.
- **Server restart**: memory is wiped on process exit. Export sessions before shutdown if you want them to survive.

If you need long-lived persistence across restarts, the export-and-restore pattern is how to build it. A small background job that periodically exports active sessions to a database, and restores them on startup, is enough for most use cases. There is no first-party database integration for this; mistralrs does not want to own that.

## What the stored state actually contains

A session holds:

- The message history, including tool calls and tool responses that the client never explicitly sent (because they were generated by the server-side tool loop).
- Any images or videos that have been attached to past messages.
- A pointer to the Python code-execution subprocess, if one exists for this session.

The Python subprocess is not part of the exportable state. If you export a session from one server and import it into another, the subprocess does not come with it: the new server will start a fresh subprocess the first time code execution happens in the restored session. Variables and imports from the old session will be gone.

For most workloads this is fine. If it matters for your use case, the model can usually rebuild Python state on request; prompt it to re-import what it needs.

## Session and the Python SDK

From the Python SDK, sessions work the same way. The `session_id` field on `ChatCompletionRequest` takes the same string, and the runner exposes `export_session`, `import_session`, and `delete_session` methods that mirror the HTTP endpoints. The [Python agentic session guide](/mistral.rs/guides/python/agentic-session/) has examples.
