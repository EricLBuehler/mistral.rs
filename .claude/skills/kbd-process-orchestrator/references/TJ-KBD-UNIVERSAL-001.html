01
Core Thesis
From code production to continuous knowledge integration — for any project
Foundational principle
Software development can be treated as continuous knowledge capture, formalization, and refinement. When that knowledge is made explicit in structured artifacts — specs, phase plans, constraint libraries, reflection reports — any capable AI agent can act on it reliably, independently of which tool happens to be running. KBD solves three root problems: session amnesia, tool fragmentation, and spec drift. The goal is Continuous Knowledge Integration, not just continuous delivery.

Traditional software delivery is skill-dependent and context-ephemeral. A developer holds knowledge in their head; when they leave, the knowledge leaves with them. AI coding assistants face the same problem at a different scale: each conversation starts cold, each tool has its own planning state, and implementation drifts from documented intent between sessions.

Knowledge-Based Development (KBD) solves this by making knowledge first-class. Architectural decisions, constraints, domain rules, and implementation history are captured in structured artifacts stored in .kbd-orchestrator/ — a file-based state directory that any AI tool can read, reason over, and act on without re-explanation.

KBD is project-agnostic and tool-agnostic. It derives project identity from existing config files (AGENTS.md, CLAUDE.md, README.md, package.json, Cargo.toml). It coordinates execution across Antigravity, Roo Code, Cursor, Claude Code, Codex, Cline, Windsurf, OpenCode, Kilo Code — any tool that can read and write files. OpenSpec is an optional enhancement, not a requirement.

The Three-Layer Knowledge Stack
Each layer informs the layer below it. All three must be maintained.
Layer 1 · Project Intelligence
Identity, Constraints, Architecture Invariants
Cross-cutting decisions and constraints that govern the entire project lifetime. Read by every agent at session start. Defines what is structurally impossible to violate. KBD auto-discovers these files — they are not authored by KBD itself.
Universal sources (read in priority order)
.kbd-orchestrator/project.json — generated by /kbd-init; project name, stack, agent config, build commands
AGENTS.md — product mission, architecture rules, never-do list
CLAUDE.md — current focus, avoid-list (when present)
README.md, package.json, Cargo.toml, pyproject.toml — fallback identity sources
.kbd-orchestrator/constraints.md — project-specific blocking + warning constraint rules
Layer 2 · Phase & Change Specifications
Phase Plans, Change Specs, Design Decisions
Feature-level knowledge baseline. Phase plans scope what is being built now. Change artifacts specify how each discrete unit of work should be implemented. Stable once created; amendments require new artifacts.
Universal sources
.kbd-orchestrator/phases/<phase>/assessment.md — gap analysis from Assess step
.kbd-orchestrator/phases/<phase>/plan.md — ordered change list from Plan step
.kbd-orchestrator/phases/<phase>/execution.md — backend dispatch contract
openspec/changes/<id>/ — structured change artifacts (if OpenSpec available)
.kbd-orchestrator/changes/<id>/change.md — native change spec (if OpenSpec absent)
Layer 3 · Execution Artifacts & Progress
Running Code, Test Results, Progress State, Reflection Reports
Concrete output of each change. Produced by AI execution agents operating under the constraints of the two layers above. Verified by artifact-refiner and bdd-testing before archiving. All progress committed to git.
Universal sources
.kbd-orchestrator/phases/<phase>/progress.json — live task ledger, updated by every executing tool
.kbd-orchestrator/current-waypoint.json — canonical resume contract across sessions and tools
.refiner/artifacts/<n>/refinement_log.md — per-change QA trail from artifact-refiner
.kbd-orchestrator/phases/<phase>/reflection.md — phase retrospective seeding the next phase
03
The PMPO Iteration Loop
The invariant structure at every level of granularity — phase, change, and artifact

Prometheus Meta-Prompting Orchestration (PMPO) is the control architecture governing how thinking, planning, execution, and reflection occur across all KBD cycles. It runs at three nested granularities simultaneously and is not a prompt template or chain-of-thought wrapper — it is a meta-level coordination loop that all KBD skills implement.

PMPO loop · invariant across macro (phase), meso (change), and micro (artifact) levels
1
Assess
Inspect current codebase and spec against stated phase goals. Surface gaps, misalignments, and technical debt. Produces a structured assessment report written to .kbd-orchestrator/phases/<phase>/assessment.md — never lost on session reset. Optionally delegates deep analysis to iterative-evolver.
2
Analyze
Scan for opportunities and highest-leverage gaps. Identify what to build next based on goal priority, risk, and dependency order. Produces the prioritized change inventory for the Plan step.
3
Plan
Produce an ordered list of discrete, implementable, verifiable changes for this phase. Each change is one slice of the product. Write to plan.md and execution.md — the backend selection and dispatch contract for execution agents.
4
Execute (via execution agent)
Select the appropriate execution backend. Dispatch each change. The executing tool reads the change spec, updates progress.json at every task boundary, and commits state to git. QA gates (artifact-refiner, bdd-testing) run per-change before archiving.
5
Reflect
After all changes in the phase are archived, run /kbd-reflect. Produces a phase reflection report capturing goal achievement, technical debt introduced, lessons learned, and seeds for the next phase. Committed to reflection.md and used as prior context for the next /kbd-assess invocation.
ℹ
State must never live in conversation
Every output of every PMPO phase — assessments, plans, execution contracts, progress ledgers, reflection reports — is written to disk and committed to git. Session resets, token limits, and agent swaps cannot lose progress because progress is never stored in context.

The Three Levels of KBD
PMPO runs at phase, change, and artifact granularity simultaneously
Level 1 · Global Phase
KBD Process Orchestrator (this skill)
Owns the Assess → Analyze → Plan → Execute (backend selection + dispatch) → Reflect loop at the phase level. Maintains canonical phase state in .kbd-orchestrator/. Delegates execution to OpenSpec, a native planner backend, or a designated AI tool.
Entry commands
/kbd-init · /kbd-assess · /kbd-plan · /kbd-execute · /kbd-reflect · /kbd-full-phase
Level 2 · OpenSpec Change (inner loop)
OpenSpec skill set (optional enhancement)
Structured change management with proposal → design → tasks → archive. OpenSpec provides a richer artifact contract between planning and execution. When OpenSpec is absent, KBD uses .kbd-orchestrator/changes/<id>/change.md with equivalent structure.
Entry commands (when OpenSpec available)
/opsx:new · /opsx:apply · /opsx:verify · /opsx:archive
Level 3 · Artifact QA (innermost)
artifact-refiner skill + bdd-testing skill
Per-change quality assurance. artifact-refiner enforces constraint-driven code quality and iterates until all blocking constraints pass. bdd-testing provides behavioral verification — Gherkin scenarios must pass before a change is marked DONE.
Entry commands
/refine-code · /refine-validate · /bdd-feature · pnpm test:bdd

Tool Registry
KBD is tool-agnostic. Any agent that can read and write files can participate.

KBD does not depend on any single AI tool's internal planning mechanism. The .kbd-orchestrator/ file system is the universal coordination contract. Any agent reads the waypoint and change spec, executes work, and updates progress.json. The table below describes the recommended role for each recognized execution agent.

Global Planner
Antigravity
Multi-file feature development with browser-based UI verification. Reads SKILL.md slash commands. Can record browser flows as BDD evidence. Preferred for phases requiring visual proof of delivery.
Phase kickoffs, UI verification, multi-file features
Architect + Executor
Roo Code
Architect mode for system-level planning and design decisions. Code mode for focused, bounded implementation tasks. Mode-switching within a single IDE session. Entry: Roo Architect or Roo Code prompt.
Architecture decisions, bounded task execution
Planner + Executor
Cursor Agent
Multi-file refactoring with parallel subagent task dispatch. Subagents update progress.json on completion. Rich file context in IDE. Team-shareable commands via .cursor/commands/.
Parallel subagent tasks, multi-file refactoring
Executor
Claude Code
Terminal-native, codebase-aware agent. Reads CLAUDE.md for constraints. Executes well-scoped changes end-to-end, runs build and test commands, fixes compilation errors iteratively. Team-shareable via .claude/commands/.
Large architectural changes, CLI-driven execution
Executor
Codex (OpenAI)
Parallel isolated tasks via git worktrees. Each change runs in its own worktree; PR or merge updates the main branch. progress.json committed in worktree and merged back. Reads AGENTS.md at repo root.
Parallel isolated tasks, PR-based delivery
Plan + Act
Cline
Terminal-first agentic workflows with Plan/Act mode separation. Plan mode reads assessment + change spec and produces a task list. Act mode executes tasks and updates progress.json per task. Invocation: /kbd-assess.md (filename includes .md suffix).
Terminal-first workflows, Plan/Act separation
Executor
Windsurf Cascade
Autonomous multi-step tasks with shared Cascade session. Flow workspace tracks progress within the session; commit progress.json at checkpoints. Workflows via .windsurf/workflows/. Team-shareable via git.
Autonomous multi-step execution, shared sessions
Executor
OpenCode / Kilo Code
Targeted single-file or few-file task execution. OpenCode: CLI/extension with scriptable commands. Kilo Code: VSCode-embedded for focused edits. Both update progress.json manually after targeted operations.
Quick targeted edits, file patches, CI/CD loops
◆
KBD is the orchestrator — tools are workers
KBD owns phase state. Execution agents report back by updating progress.json and committing it to git. This gives every tool cross-session and cross-tool visibility into what has been done, what is blocked, and what comes next — without any special runtime or message bus.
06
File-Based State System
.kbd-orchestrator/ is the universal coordination contract

KBD uses .kbd-orchestrator/ as its single source of truth because every AI tool can read and write files. Git provides version history, merge conflict detection, and cross-tool synchronization. No special runtime, message bus, or API integration is required. Any tool that can git pull and write files can participate in a KBD process.

.kbd-orchestrator/current-waypoint.json
Written by any orchestrator · Read by all tools
Canonical resume contract. Contains active phase, last completed change, next pending change, exact next command, and fallback command. Load this before anything else on every session start.
.kbd-orchestrator/project.json
Generated by /kbd-init · Read by all tools
Project identity + configuration. Name, stack, build/test/lint commands, preferred agents, OpenSpec availability flag. Never commit project-specific values into the skill directory itself.
.kbd-orchestrator/phases/<phase>/assessment.md
Written by /kbd-assess · Read by /kbd-plan
Structured gap analysis output. Lists what is implemented vs. what is missing relative to phase goals. The factual basis for the change plan.
.kbd-orchestrator/phases/<phase>/plan.md
Written by /kbd-plan · Read by /kbd-execute
Ordered change list for this phase. Each entry is one discrete, implementable, verifiable slice. Read-only once created — amendments require a new phase plan.
.kbd-orchestrator/phases/<phase>/execution.md
Written by /kbd-execute · Read by all tools
Backend dispatch contract. Specifies which agent handles which change, entry points, and special instructions per change. The handoff document between orchestrator and executor.
.kbd-orchestrator/phases/<phase>/progress.json
Updated by every executing tool · Read by /kbd-status
Live task progress ledger. Every tool that starts, makes progress on, or completes a change MUST update this file and commit it to git. The universal cross-tool heartbeat.
.kbd-orchestrator/phases/<phase>/reflection.md
Written by /kbd-reflect · Read by next phase /kbd-assess
Phase retrospective. Goal achievement delta, debt introduced, lessons, next phase suggestions. Passed as prior_assessments to the next evolver invocation — the primary mechanism for compounding knowledge across phases.
.kbd-orchestrator/constraints.md
Authored by human or /kbd-init · Read by all QA tools
Project-specific blocking and warning constraint rules. Single source of truth for what violates the constraint envelope. Both artifact-refiner and executing tools read this file — never define constraints in two places.
07
Wayfinding Protocol
How KBD resumes cleanly across sessions, tools, and interruptions

The most common failure mode in multi-session AI development is not execution quality — it is re-orientation cost. Each new session must re-discover where work left off, which changes are done, what is blocked, and what comes next. KBD eliminates this with a mandatory wayfinding artifact that any tool can read to resume instantly.

current-waypoint.json — minimum required fields
Written after every phase state transition. Load this before any other action on session start.
Canonical waypoint structure
"active_phase": "phase-2-sales-module"
"backend": "antigravity"
"last_updated": "2026-03-10T14:22:00Z"
"last_updated_by": "roo-code"
"last_completed_change": "change-007-team-invitations"
"next_pending_change": "change-008-clients-page"
"preferred_re_entry_skill": "kbd-process-orchestrator"
"exact_next_command": "/kbd-execute phase-2-sales-module"
"fallback_command": "/kbd-status — if execution.md is stale, re-run /kbd-plan"
ℹ
Waypoint-first resumption is non-negotiable
On every new session, the first action is cat .kbd-orchestrator/current-waypoint.json. This file tells you exactly where to resume. Do not infer status from broader phase discovery until you have confirmed the waypoint is stale or absent.
08
Cross-Tool Handoff Protocol
The progress.json contract that every executing tool must follow

When an AI tool is dispatched to execute a KBD change, it must follow a strict protocol for reading and updating progress.json. This file is the heartbeat of the KBD process — the universal mechanism by which any tool can know what any other tool has done.

On Change Start
Read waypoint + change spec
Pull latest git state. Read current-waypoint.json, then the change spec. Update progress.json status → IN_PROGRESS, started_by → your tool name. Commit.
On Each Task Complete
Increment tasks_done, commit
Update tasks_done, last_task_completed, next_task_pending. Commit every few tasks: git add .kbd-orchestrator && git commit -m "kbd: progress [tool] change-id task N/M"
On Change Complete
Archive + advance waypoint
Status → DONE. Archive (OpenSpec or native KBD). Advance waypoint last_completed_change and next_pending_change. Final commit. Echo [kbd] Change complete.
On Blocker
Surface blocker, halt
Status → BLOCKED. Add to blockers array. Update waypoint fallback_command with specific description of what is blocking. Commit. Do not attempt to resolve autonomously.
Conflict Resolution
Merge progress.json manually
Never auto-resolve to "ours" or "theirs." Take the highest tasks_done count, most recent timestamp, union of blockers. Status priority: BLOCKED > IN_PROGRESS > DONE > PENDING.
Never Do
Forbidden operations
Do not delete or replace progress.json wholesale. Do not reset tasks_done. Do not change started_by once set. Do not modify plan.md during execution. Do not skip git commits.
09
iterative-evolver
Deep assessment and phase lifecycle engine — global skill at .agent/skills/iterative-evolver/
Integrated Skill iterative-evolver · Phase Assessment & Reflection Engine
The iterative-evolver skill owns deep codebase gap analysis and structured phase reflection with cross-session continuity. KBD's lightweight /kbd-assess is the default; when a phase requires multi-dimensional analysis or continuity across many sessions, KBD delegates to iterative-evolver. State backend: .evolver/evolutions/<evolution-name>/state.json — independent from KBD state, never in conversation.
When KBD delegates to iterative-evolver
1
For deep phase assessment: invoke /evolve-assess "<project>-<phase>" with evolution_domain: software, phase goals, target state, project path, and prior phase reflection as prior_assessments. The evolution name is the cross-session resume key — store it in project.json as evolver_evolution_name.
2
The skill produces a structured assessment report written to .evolver/evolutions/<evolution-name>/reports/. KBD copies or symlinks this report to .kbd-orchestrator/phases/<phase>/assessment.md as the canonical gap analysis for the Plan step.
3
For phase reflection: invoke /evolve-report "<evolution-name>" after all changes in the phase are archived. The output becomes reflection.md in the KBD phase directory and is passed as prior_assessments to the next phase's evolver invocation — this is how the system compounds knowledge across phases.
When NOT to use iterative-evolver
✕
For small phases with fewer than 3 changes, use /kbd-assess directly — it is faster and sufficient. Use iterative-evolver when deep multi-dimensional analysis or cross-session assessment continuity is required.
10
artifact-refiner
Change-level quality assurance engine — global skill at .agent/skills/artifact-refiner/
Integrated Skill artifact-refiner · Constraint-Driven Code QA Gate
The artifact-refiner skill owns constraint-driven code quality enforcement at the artifact level. KBD invokes it per completed change — after the executing tool marks the change DONE in progress.json, and before /opsx:archive or native KBD archiving. Constraints are always sourced from .kbd-orchestrator/constraints.md, never defined independently. State backend: .refiner/artifacts/<n>/.
KBD → Refiner lifecycle
1
After a change is implemented, invoke /refine-code "<change-id>" with artifact_type: code, content_type: direct:code, and the project constraint list sourced from .kbd-orchestrator/constraints.md. Wire on_iteration_complete to run the project's build health command from project.json.
2
The skill iterates — using constraint violations as the driver for refinement — until all blocking constraints pass or maximum iterations (5) are reached. Each iteration writes results to .refiner/artifacts/<change-id>/refinement_log.md.
3
On convergence, KBD reads the refinement log. If any blocking constraint remains FAIL, the change is marked BLOCKED in progress.json. If all blocking constraints pass, proceed to /opsx:verify and archive. The refinement log is committed to the repo as the audit trail for that change.
Constraint wiring principle
◆
.kbd-orchestrator/constraints.md is the single source of truth for constraints. Never define constraints independently in the refiner invocation. The refiner uses KBD's own constraint file, eliminating duplication between orchestrator and QA gate.
11
bdd-testing
Behavioral verification gate — global skill at .agent/skills/bdd-testing/
Integrated Skill bdd-testing · Scenario-Driven Acceptance Verification
The bdd-testing skill provides behavioral verification — Gherkin scenarios written from the user's perspective that must pass before a change is marked DONE. BDD feature files serve as living specifications that prove functional completeness, not just structural presence. Named after the change ID for traceability: tests/features/<layer>/<change-id>-<description>.feature.
Placement within a KBD change lifecycle
1
Plan step: When creating the change spec, generate Gherkin scenarios as acceptance criteria. These are the definition of "done" from the user's perspective before a line of code is written.
2
Execute step: After the executing tool implements the feature, write step definitions and run pnpm test:bdd:tag "@<change-id>". All tagged scenarios passing = change DONE. Exit code non-zero = change stays IN_PROGRESS, blockers updated.
3
Reflect step: BDD coverage report informs the goal completion percentage in reflection.md. For Antigravity-executed phases, browser recording proofs can be embedded as video evidence links in the delivered changes section.
When NOT to use bdd-testing
✕
For infrastructure-only changes (migrations, config) with no UI or API surface, skip BDD and use direct integration test commands. For changes under 30 minutes complexity, a Playwright script may be more efficient than Gherkin. BDD is most valuable for user-facing features where acceptance criteria can be stated from the user's perspective.

pmpo-skill-creator
KBD self-improvement engine — global skill at .agent/skills/pmpo-skill-creator/
Integrated Skill pmpo-skill-creator · Meta-Skill Extension via Reflection
The pmpo-skill-creator skill enables KBD to extend itself. During the Reflect phase, if a structural gap is identified — a recurring workflow that KBD does not yet handle, a new integration pattern, or a domain-specific sub-skill needed for future phases — /create-skill or /extend-skill is invoked to formalize the new capability into the skill system. This is how KBD grows smarter across projects and phases.
When KBD invokes pmpo-skill-creator
1
During /kbd-reflect: if the reflection report identifies a recurring workflow gap (e.g., "every phase requires a database migration verification step we don't have a skill for"), invoke /create-skill with the observed gap as input.
2
If an existing skill (iterative-evolver, artifact-refiner, etc.) needs a new integration adapter for a new tool or pattern, invoke /extend-skill with the extension specification.
3
New skills are placed in .agent/skills/ and referenced from KBD's integration layer, maintaining the single-source-of-truth principle. The skill creator's output feeds back into the next phase's KBD toolchain.

How the four global skills interlock within the KBD phase loop

The four integrated skills are not copies inside the KBD skill directory — they live in the global .agent/skills/ directory and are invoked by reference. This preserves a single source of truth for each skill, ensures updates propagate automatically, and avoids maintenance split. Integration adapters in references/integrations/ specify the exact invocation contract without duplicating any logic.

OpenSpec Layer
Optional structured change management — KBD adapts when OpenSpec is absent

OpenSpec is a structured change management system that provides a richer artifact contract between planning and execution. It is an enhancement to KBD, not a dependency. KBD detects OpenSpec availability from the presence of an openspec/ directory and adapts its change management approach accordingly.

When OpenSpec IS available
Structured change artifacts with full lifecycle
Use /opsx:new to scaffold proposal → design → tasks → spec-delta. Progress tracked in openspec/changes/<id>/tasks.md via checkbox protocol. Archiving via /opsx:archive syncs spec deltas back to canonical specs and feeds the reflection phase.
OpenSpec artifact contract
openspec/specs/*.md — canonical spec baseline; what the system does
openspec/changes/<id>/proposal.md — scoped intent: what, why, how
openspec/changes/<id>/design.md — entities, ports, adapters, file layout
openspec/changes/<id>/tasks.md — concrete implementation checklist
openspec/changes/<id>/spec-delta.md — how canonical specs change after this change
When OpenSpec is NOT available
Native KBD change management
Create .kbd-orchestrator/changes/<id>/change.md with the same structure as a combined proposal + tasks document. Track task status with [ ] / [/] / [x] checkboxes. Archive by moving to .kbd-orchestrator/changes/archive/<date>-<id>/. The execution.md format accommodates both paths.
Native KBD change structure
.kbd-orchestrator/changes/<id>/change.md — proposal + tasks combined
.kbd-orchestrator/changes/archive/<date>-<id>/ — finalized changes
◆
KBD never requires OpenSpec
The decision to use OpenSpec is a project configuration choice, not a KBD requirement. Set openspec_available: true in .kbd-orchestrator/project.json to enable OpenSpec paths. All KBD commands adapt their artifact handling based on this flag.
15
Constraint System
Project-derived rules that define the quality envelope — not hard-coded, not duplicated

Unlike the earlier DocuMind-specific reference, KBD does not hard-code any project's constraints. Constraints are always defined in .kbd-orchestrator/constraints.md at the project root — authored by the human developer or generated by /kbd-init from a generic template. Both artifact-refiner and all executing tools read this single file. Constraints are never defined in two places.

Generic constraint categories · source: .kbd-orchestrator/constraints.md
blockingBuild passes without errors — command specified in project.json as build_health_command
blockingNo hardcoded secrets, API keys, tokens, or passwords in source code
blockingArchitecture invariants defined in AGENTS.md "Never Do" list — e.g., no cross-layer imports, no direct provider calls
blockingNo runtime type-unsafe patterns (varies by stack — unwrap() in Rust, any in TypeScript, etc.)
blockingProject-specific structural invariants (defined per project in .kbd-orchestrator/constraints.md)
warningTests exist for all new features added in this change
warningNo lint warnings at default severity — command specified in project.json as lint_command
warningNo TODO/FIXME/STUB/HACK comments in committed code
warningAccessibility basics for new UI components (aria-label or semantic HTML)
✓
Constraints are the convergence engine
The refiner iterates as long as blocking constraints remain violated. A change is never archived with known constraint violations — regardless of which execution agent produced the code. The constraint file, not the agent, is the source of truth for "done."

Command Reference
The complete KBD command surface — available across all supported tools
/kbd-init [--force] [--dry-run]Auto-discover project identity and generate .kbd-orchestrator/project.json. Runs the command generator script for all configured tools. Run first in any new project.
/kbd-new-phase <name> [goals...]Start a new named phase with optional goal statements. Creates the phase directory structure in .kbd-orchestrator/phases/<name>/.
/kbd-assess [phase-name]Assess current codebase against active phase goals. Lightweight wrapper — delegates to iterative-evolver for deep multi-dimensional analysis when needed. Writes assessment.md.
/kbd-plan [phase-name]Create prioritized change list from the current assessment. Writes plan.md and execution.md. Read-only once created — amendments require a new plan iteration.
/kbd-execute [phase-name]Select execution backend from the tool registry and dispatch the phase. Writes or updates execution.md. Updates waypoint with exact next command for the selected backend.
/kbd-reflect [phase-name]Generate phase reflection report from current progress state. Delegates to iterative-evolver for structured reflection. Writes reflection.md and seeds the next phase's prior_assessments.
/kbd-statusShow current phase, change inventory with per-change status, and waypoint-guided next action. Reads current-waypoint.json and progress.json. Primary orientation command on session start.
/kbd-full-phase <name>Run the complete Assess → Plan → Execute → Reflect cycle in one invocation. For experienced use on well-understood phases. Pauses for human approval at the Plan → Execute boundary.
/opsx:new <change-name>Scaffold a new OpenSpec change with proposal, tasks, design, and spec-delta artifacts. (Requires OpenSpec to be available in the project.)
/opsx:apply [change-name]Implement tasks from an OpenSpec change. Loops through the checklist until complete or blocked. Dispatches to the configured execution backend.
/opsx:verify [change-name]Verify implementation against change artifacts: completeness, correctness, coherence. Three-dimension QA check before archiving.
/opsx:archive [change-name]Finalize and archive a completed change. Syncs spec deltas to canonical specs. Updates progress.json and advances the waypoint.
17
Cross-Tool Command Matrix
KBD commands are generated from a single canonical YAML source — never edited directly per tool

KBD uses a single workflows/workflows.yaml source and a bash generator script (scripts/kbd-generate-commands.sh) to produce tool-specific command files. All tools share the same underlying PMPO patterns — only the file location and invocation syntax differ. After any change to workflows.yaml, run the generator to sync all tool directories.

The Path to Autonomous Development
How a maturing KBD knowledge base reduces human overhead per phase

As the KBD knowledge base matures — richer phase reflections, deeper constraint libraries, more complete spec baselines, broader skill coverage — the workflow trends naturally toward autonomy. The bottleneck is never execution capability. Current AI agents can already write production-quality code. The bottleneck is knowledge quality. A well-maintained KBD stack directly accelerates the path.

Now
Human-Directed
Human drives all planning decisions. KBD structures the assessment; human approves the plan before execution begins. AI executes against change specs. Human reviews refiner output and reflection reports before the next phase begins.
Near-Term
AI-Proposed
KBD and iterative-evolver propose phases autonomously based on accumulated reflection history. Human approves the plan at the Plan → Execute boundary. Execution and QA (artifact-refiner + bdd-testing) run without per-change interruption.
Mid-Term
Supervised Loops
KBD detects spec gaps, creates change specs, dispatches execution agents, routes constraint failures back through the QA gate, and archives completed changes — all without human intervention per change. Human reviews phase reflections and approves phase transitions.
Long-Term
Strategic Oversight
Knowledge base rich enough that new features synthesize from domain knowledge alone. The system identifies analogous prior implementations and adapts them. Human oversight concentrates on high-level product direction and architectural invariant governance.
Key insight — universal to all projects
The bottleneck to full autonomy is not execution capability — current AI agents can already write production-quality code across any stack. The bottleneck is knowledge quality. A well-maintained KBD stack — phase reflections, constraint libraries, canonical specs, cross-tool progress state — directly accelerates the path to autonomous, self-improving software delivery. Every phase you run builds the knowledge base that makes the next phase easier.
